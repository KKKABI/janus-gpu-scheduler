## 一、方向1进展回顾（HP-HP Interference 优化）

### 已做的工作

1. 用**多维资源向量余弦相似度**替代 janus 原有的 name-based compute/memory 二分类
   - 每个算子：4 维向量 (registers, shared_memory, threads, duration)
   - 候选组合内所有算子两两算余弦相似度，取均值
   - 在占用率 ≥ 0.9×max 的候选组合中，选相似度最低的（= 资源最互补）

2. 测了一组数据，与 baseline（原有 name-based 二分类）对比，无显著变化

### 被指出的问题

1. **理论依据不足**：为什么三维向量夹角小就等于资源互补/干扰小？需要讲清楚数学原理
   - 更朴素的方法：直接把候选组合里各算子的 register、shared memory、threads 分别加起来，看哪个组合超限最少
   
2. **根本性问题——候选组合太少**：卡 0.9 占用率阈值时，绝大多数情况只剩 **1 个**候选组合
   - 改了 0.8 测一遍也还是很少
   - 只有一个候选时，后续任何选择策略都没有意义
   
3. **根本短板**：当前三个指标（registers, shared memory, threads）**无法反映 L2 cache 和 memory bandwidth 的竞争**
   - 不知道每个算子取数据的量——这是 memory 干扰的核心来源
   - 用 Nsight Compute 可以测（L2 cache hit rate, DRAM bandwidth），但开销远大于当前的 1-2 秒 profiling

### 确定的下一步

1. **先统计候选组合数量**：测不同阈值下（0.9, 0.8, 0.5, 0.2）有多少候选组合可选
   - 这些数据可以直接放进大论文当 motivation
2. 候选组合数量够了，再谈选择策略才有意义
3. 如果阈值放宽到 0.5 甚至更低还是很少，需要重新审视 HP/LP 分类算法本身
4. **方向1是四个方向里最明确可做的，优先推进**

## 二、方向2/3（LP 分级 / 多优先级）

### 已做的工作

将 LP 分为两类：
- **LP-light**：CSS < 0.5 且 duration < 阈值 → 跟 HP 同轮并行，卡缝执行
- **LP-heavy**：CSS < 0.5 且 duration ≥ 阈值 → 推迟到后续执行

测了也无显著效果。

### 被指出的问题

1. **方向2和方向3本质是一件事**——都是 LP 的精细化处理
2. **"轮次"的概念在运行时不存在**：
   - HP 之间有同步 barrier，可以称为"一轮"
   - 但 LP 完全是硬件自由调度——不存在"这一轮可以捡漏、下一轮就不能"的区别
   - LP-heavy 推迟到"下一轮"，在运行时跟当前轮没区别
3. **真正的痛点**：LP 在 HP 快结束时才开始执行，跨过同步 barrier 阻塞下一组 HP，这是**运行时行为**，静态分析难以解决
4. **纯多优先级细分**（不加同步）大概率没用。加了同步又可能比不加更差
5. 导师建议先问大模型找思路，暂不投入精力

### 决定：**方向2/3 暂时搁置**

## 三、方向4（多 CUDA Graph 并发）

### 已做的工作

1. 复制 `multi_client_compare.py` 和 `scale_clients.py` 到 janus
2. 修复 janus 缺失的 `Critical_node.mark_critical_nodes()` 调用
3. 跑了 MobileNetV2 + ResNet50 的双客户端对比：
   - 所有组合 concurrent vs sequential 增益仅 1.00x~1.02x（噪声范围）
   - 单个 Opara CUDA Graph 已吃满 GPU，并发无额外收益
4. 跑了 ConvNeXt 的扩展性测试（N=1,2,4,8），随并发数增加延迟下降

### 被指出的问题

1. **统计口径不一致**：
   - pair 比较中"总延迟"：串行 = 两个加和，并发 = 取 max
   - scale 测试中"总延迟"：始终取 max（或 avg）
   - **不能用同一个名字"总延迟"表达不同含义**

2. **Profile 用的是整 GPU 信息**：两个并发推理时，profile 仍假设独占全部 SM
   - 正确做法：根据并发数给出对应份额的资源信息（如 2 并发 → 半个 GPU），让 Janus 根据可用资源选不同的 CUDA Graph
   - 这才是"按可用资源选不同 CUDA Graph"的核心思想

3. **单推理未吃满 GPU 的模型才有并发收益**：ConvNeXt（并行度 2.5）有效果，MobileNetV2 和 ResNet50 没有

### 摸底实验建议

1. 修正统计口径（区分"单个推理延迟"和"完成 N 个推理的总 wall-clock 时间"）
2. 试 NASNet（论文中并行度最高，3.1）
3. 试改 profile 资源信息（N 并发 → 1/N GPU 资源）
4. 试加大 batch size
