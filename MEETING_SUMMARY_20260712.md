## 四个研究方向

老师梳理了当前可以推进的 **四个方向**：

---

### 方向一：HP-HP Interference（老师最推荐）

**问题：** 多个 HP 算子之间的相互干扰（计算资源争抢）导致并行加速比不如预期。

**老师意见：**
- 这是四个方向里**工作量最能体现、最有文章可做**的
- 之前测出来的 max vs average 时间差图可以直接用
- 可以设计一套完整的策略体系：

```
1. 如何通过静态 profile 量化 interference
   - 用 cache、memory、ALU 等维度建模
   - 模拟两个 HP 算子放一起时的干扰程度

2. 基于 interference 的调度策略
   - 选 HP 时考虑干扰
   - 选什么 LP、怎么跟 HP 搭配

3. 能体现一套算法 + 策略 + 实验验证
```

**优先级：P0，先做这个。**

---

### 方向二：HP 本来能并行但没并行起来（LP 阻塞）

**根因确认：**

```
场景:
  第 N 轮: HP_a + HP_b + LP_large 同时发射
  HP_a 和 HP_b 跑完后
  LP_large 还在跑，占了部分 SM 资源
  第 N+1 轮 HP_c + HP_d 想并行 → 资源不够 → 串行

不是 DRAM 带宽的问题。
纯粹是 LP 占着 SM 不放。
```

**老师想法：**
- 这个问题**可以观察到**，但**解决方案我还没想好**
- 需不需要加同步？能不能做？有没有做的价值？
- 可以先问张一峰意见，但**要以自己测试为准**（"张一峰说少不一定准"）
- **你自己下去测一下，看这个问题发生的概率有多大**

---

### 方向三：多优先级细分（可发小论文）

**核心思想：** 当前只有 HP 和 LP 两种优先级，可以进一步细分。

```
当前:
  HP → 最高优先级 CUDA stream
  LP → 普通/默认优先级

可优化:
  HP 内部分:
    - 关键路径 HP → 最高优先级
    - 非关键路径 HP → 次高优先级

  LP 内部分:
    - 小 LP（短算子）→ 次低优先级，优先卡缝
    - 大 LP（长算子）→ 最低优先级，慢点执行

  硬件层面:
    多个不同优先级的 CUDA stream
    对应"HP 能并行、小 LP 卡缝、大 LP 不霸占"
```

**老师原话：**
> "这个方向，这个小的东西是可以发一篇小论文的"

**注意：** 跟方向一可以结合——Interference 小的组合放同一优先级、Interference 大的分到不同优先级。

---

### 方向四：多 CUDA Graph 并发推理

**核心思想：** 一个 GPU 上同时跑多个 Janus 推理请求。

```
场景：
  GPU 上同时跑 2/3/4 个推理（同模型 or 不同模型）
  每个推理都用 Janus 的 CUDA Graph
  不做调度，直接扔上去跑

目的：
  先跑 baseline，看多请求并发时有什么问题
  发现问题了再想怎么解决
```

**老师强调：**
- **跟 Orion 没有关系**（Orion 是 kernel 级调度）
- 先不要想调度，先**跑 baseline 看问题**
- 用 Orion 的框架但不要做任何调度
- 只测每个请求的平均延迟
- 看到问题再想：比如"选不同的 CUDA Graph 发射"
- 这个点比较小，但可以放毕业论文当一章
